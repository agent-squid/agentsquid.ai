How Squid Is Different

Squid is not another general AI chat app. Open WebUI and LibreChat are broad self-hosted chat platforms for many providers, RAG, plugins, users, memory, and web search. Squid is narrower: it controls real local coding-agent CLIs and preserves their native session behavior.

Most chat UIs send messages to a model API and render text back — the chat app is the runtime. Squid is different: the runtime is still the local CLI agent. Claude Code, Codex, Cursor Agent, OpenCode, and Pi are the processes doing the work on your machine. Squid is the interactive control layer around them.

What that means in practice

  • Real CLI sessions, not copied chat history — Squid uses the agent's native session instead of replaying a transcript.
  • Working-directory awareness — sessions are tied to the cwd the CLI actually runs in, which matters for local project context and resume behavior.
  • Process ownership — Squid can show live processes, kill the exact running job, drain queued jobs, and recover after disconnects.
  • Topic and agent lanes — #topic@agent is closer to a named terminal workspace than a chat room, though you can filter it to look like one.
  • Progress while work happens — a thought bubble surfaces status, queued state, tool activity, and partial output before the final answer lands.
  • Analytics attached to real work — token usage is tied to each prompt and rolls up by topic, agent, or time.

Is it a good fit?

Good fit if you:

  • Run multiple coding-agent terminals at once
  • Want to recover and operate sessions by name
  • Use a local Mac mini, workstation, or always-on machine for agent work
  • Want phone/tablet control without moving execution to the cloud
  • Compare or combine several agent CLIs
  • Care about token usage, cost, and per-topic/per-agent analytics

Overkill if you:

  • Only ever use one agent in one terminal
  • Want a hosted team chat product
  • Want Squid to be the coding agent itself
