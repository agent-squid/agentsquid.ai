Squid Flow

A route isn't limited to one agent. Add a hop operator after a #topic@agent and Squid dispatches the next step server-side — the handoff, round trip, or join happens whether or not your browser tab is still open.

One-way handoff

Send one agent's output straight into another agent's prompt:

#squid@codex>@review!Codex's output becomes the prompt for a fresh @review session.

Round trips

Send a message, get a response, and send that response back — for one or more rounds:

#squid@codex<>@review!One round trip: codex to review, then review's answer back to codex.
#squid@codex<3>@review!Three round trips in a row.

Joins

Run more than one step in parallel, then combine their output into a single next step with +:

#t1@a+#t2@b>#t3#t3 runs once, after both #t1 and #t2 finish, with both outputs available to it.
#t1@a1<>#t2+#t3#t2 and #t3 run in parallel from #t1's output; once both finish, one turn goes back to #t1 with both.

Scheduled handoffs

Space repeated one-way handoffs out over time:

#squid@codex=3:1d>@reviewThree independent handoffs to @review, one per day.

Fresh vs. persistent sessions

Add ! to any step in a chain to run it as a fresh native session instead of continuing that agent's persistent lane — the same ! you'd use outside a chain.

This page covers the syntax you'll actually use day to day. For the full grammar, resolution rules, and execution model — useful if you're extending Squid itself — see the Squid Flow whitepaper in the GitHub repo.
